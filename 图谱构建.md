# 图谱构建

本文描述代码知识图谱在工程中的**构建与更新**路径，聚焦 **全量索引** 与 **增量索引** 的领域编排、与 Git / PostgreSQL / Neo4j / 向量库的交互，以及触发入口的差异。实现以 `evaluation-domain` 的索引与图谱服务为主，`evaluation-trigger` 提供 HTTP、调度与消息消费入口。

---

## 1. 触发方式总览

| 触发源 | 入口类 | 调用方式 | 索引类型 | 说明 |
|--------|--------|----------|----------|------|
| 手动 HTTP | `IndexController` | `indexTaskExecutor` 异步执行 `FullIndexService.runFullIndex` | 全量 | `POST /api/v1/index/trigger`，立即返回 `SUBMITTED`。 |
| 定时调度 | `IndexScheduler` | 调度线程**同步**逐个仓库调用 `runFullIndex` | 全量 | Cron 默认每天 02:00，可配置 `index.scheduler.cron`；单仓失败记录日志后继续下一仓。 |
| GitHub Webhook | `GithubWebhookController` → `GithubPushWebhookService` | Tomcat 线程**同步**执行增量编排 | 增量 | 仅 `X-GitHub-Event: push`；验签、对齐 `clone_url` 后执行。 |
| Kafka（可选） | `IndexConsumer` | 监听线程消费后**同步**调用 `IncrementalIndexOrchestrator.run` | 增量 | Topic 默认 `code-change-event`，与 Webhook 共用同一编排器。 |

**图谱写入的统一出口**：成功解析后的批量落库经 `CodeGraphService.batchWriteParseResults` → `GraphAdapter`（Neo4j 实现为 `Neo4jGraphAdapterImpl`）。全量路径在达到批阈值时还会调用 `EmbeddingService.submitAsync`（异步向量写入）；增量路径在**删除文件**时会清理向量，**新增/修改**当前编排器内**未**调用异步 Embedding 提交（与全量行为不一致，以代码为准）。

## 2. 全量索引：领域流程与时序

### 2.1 职责与数据流（文字）

`FullIndexServiceImpl.runFullIndex` 顺序概要：

1. **PostgreSQL**：`RepositoryPort` 按 `repoId` 取仓库；不存在则抛错。
2. **PostgreSQL**：`IndexTaskPort` 创建 `FULL` 任务，`RUNNING`。
3. **Git**：`GitAdapter.cloneOrPull` 更新本地工作副本。
4. **Neo4j**：`CodeGraphService.writeRepositoryNode` 写入/合并 `Repository` 节点。
5. **本地磁盘**：遍历仓库下所有 `**/src/main/java` 目录作为 `sourceRoots`；`FileWalkerService.walkJavaFiles` 收集全部 `.java` 路径。
6. **逐文件循环**：
   - 用 `FileWalkerService.computeChecksum` 与 `CodeGraphService.getFileChecksum`（读 Neo4j 中已有 `JavaFile.checksum`）比较，相同则**跳过解析**（幂等）。
   - `JavaAstParserService.parse` 生成 `ParseResult`；失败则 `ParseErrorPort.record` 并跳过该文件。
   - 将多个文件的解析结果放入内存缓冲；当缓冲内「类型+方法+字段」节点数累计 ≥ 500 时：
     - `CodeGraphService.batchWriteParseResults` → Neo4j（先 Package / JavaFile / Type / Method / Field 节点，再分批关系）。
     - `EmbeddingService.submitAsync` → `@Async("embeddingExecutor")` 的 `EmbeddingSubmitter` 对 `Type`/`Method` 调用 `EmbeddingStoreAdapter.upsertEmbedding`。
7. 循环结束后 **flush** 剩余缓冲（再次写 Neo4j + 提交异步 Embedding）。
8. **PostgreSQL**：任务状态更新为 `SUCCESS`；异常则 `FAIL` 并抛出 `IndexTaskException`。

### 2.2 时序图（核心参与方）

下列时序图省略「逐文件循环」内部的反复消息，用 **loop** 表示；Neo4j 与 PG 分别抽象为存储参与方。

```mermaid
sequenceDiagram
    autonumber
    participant T as 触发方<br/>(调度器 / 线程池)
    participant F as FullIndexServiceImpl
    participant PG as PostgreSQL<br/>(RepositoryPort / IndexTaskPort / ParseErrorPort)
    participant G as GitAdapter
    participant C as CodeGraphService
    participant N as Neo4j<br/>(经 GraphAdapter)
    participant W as FileWalkerService
    participant J as JavaAstParserService
    participant E as EmbeddingService<br/>(→ EmbeddingSubmitter 异步)

    T->>F: runFullIndex(repoId)
    F->>PG: findById(repoId)
    PG-->>F: RepositoryInfo
    F->>PG: createTask(FULL, RUNNING)
    PG-->>F: IndexTask(taskId)

    F->>G: cloneOrPull(url, branch, localPath)
    G-->>F: ok

    F->>C: writeRepositoryNode(...)
    C->>N: MERGE Repository

    F->>F: findSourceRoots(仓库路径)
    F->>W: walkJavaFiles(repoPath)
    W-->>F: 全部 .java 路径列表

    loop 每个 .java 文件
        F->>W: computeChecksum(path)
        F->>C: getFileChecksum(path)
        C->>N: 读取 JavaFile.checksum
        N-->>C: stored
        C-->>F: stored
        alt checksum 未变化
            F->>F: continue（跳过解析）
        else 需解析
            F->>J: parse(path, sourceRoots, repoId)
            alt 解析失败
                J-->>F: ParseResult(success=false)
                F->>PG: record(PARSE_ERROR, ...)
            else 解析成功
                J-->>F: ParseResult
                F->>F: 缓冲；累计节点数 ≥ 500 则 flush
                opt flush 批
                    F->>C: batchWriteParseResults(batch)
                    C->>N: MERGE 节点与关系（分批）
                    F->>E: submitAsync(batch, repoId)
                    Note over E: @Async 线程池内向量 upsert
                end
            end
        end
    end

    opt 缓冲非空
        F->>C: batchWriteParseResults(remainder)
        C->>N: MERGE
        F->>E: submitAsync(remainder, repoId)
    end

    F->>PG: updateTaskStatus(taskId, SUCCESS)
```

### 2.3 手动触发与定时触发的差异

- **手动**（`IndexController`）：HTTP 线程只把 `runFullIndex` 提交到 `indexTaskExecutor`，**立即**返回；实际全量在后台线程执行，时序图触发方写为「线程池中的任务」即可。
- **定时**（`IndexScheduler`）：`@Scheduled` 方法内**顺序**对每个 ACTIVE 仓库调用 `runFullIndex`，与上图一致但无单独线程池包裹单次调用。

## 3. 增量索引：领域流程与时序

### 3.1 职责与数据流（文字）

`IncrementalIndexOrchestratorImpl.run(CommitEvent)` 顺序概要：

1. **幂等**：`CommitRecordPort.isProcessed(repoId, commitHash)` 为真则直接返回 `skippedAlreadyProcessed=true`，**不写图**、不更新任务表（与 Webhook 中跳过 Dify 的分支一致）。
2. **PostgreSQL**：校验仓库存在；`IndexTaskPort` 创建 `INCREMENTAL` 任务 `RUNNING`。
3. **Git**：`cloneOrPull` 后 `fetch`，保证能对比到父提交。
4. **Diff**：若 `parentCommitHash` 为 GitHub 全 0 空 SHA 等「空父」情形，则用 `diffCommitAgainstFirstParent`；否则 `diffCommits(parent, after)`，得到变更 Java 文件列表；并计算 `diffLineStats` 供统计与下游证据。
5. 若无变更 Java 文件：任务置 `SUCCESS` 并返回（**不**写入 `Commit` 节点）。
6. **逐变更文件** `processChangedFiles`：
   - **DELETED**：列出该文件下图谱节点的向量键 → `EmbeddingStoreAdapter.deleteEmbeddingsByNodeIds` → `GraphAdapter.deleteFileNodes`（级联删 Method/Field/Type/JavaFile）。
   - **MODIFIED**：`GraphAdapter.deleteFileOutgoingRelations` 清除该文件相关出边，再全量重解析该文件并 MERGE 回图。
   - **ADDED** 等其余类型：直接解析并稍后批量写入。
   - 解析失败：`ParseErrorPort.record`，该文件不进入成功结果集。
7. 若存在任一成功解析结果：`CodeGraphService.batchWriteParseResults` 一次性写入本批 `ParseResult`（增量按变更文件集合，通常单批即可）。
8. **Neo4j**：`GraphAdapter.batchMergeCommitNodes` 写入本次 `Commit` 节点（**当前实现未**随此步骤写入 `CHANGED_IN` 边，仅以 `Commit` 元数据落库；查询侧若依赖 `CHANGED_IN` 需与实现演进对齐）。
9. **PostgreSQL**：`CommitRecordPort.markProcessed`；`IndexTaskPort` 更新 `SUCCESS`。异常则任务 `FAIL` 并抛 `IndexTaskException`。

### 3.2 时序图（单文件分支合并描述）

```mermaid
sequenceDiagram
    autonumber
    participant S as 调用方<br/>(Webhook / Kafka)
    participant O as IncrementalIndexOrchestratorImpl
    participant CR as CommitRecordPort
    participant PG as PostgreSQL<br/>(Repository / IndexTask / ParseError)
    participant G as GitAdapter
    participant P as JavaAstParserService
    participant GA as GraphAdapter
    participant N as Neo4j
    participant C as CodeGraphService
    participant ES as EmbeddingStoreAdapter

    S->>O: run(CommitEvent)
    O->>CR: isProcessed(repoId, hash)
    alt 已处理
        CR-->>O: true
        O-->>S: IncrementalIndexResult(skippedAlreadyProcessed)
    else 未处理
        CR-->>O: false
        O->>PG: findById / createTask(INCREMENTAL, RUNNING)
        O->>G: cloneOrPull + fetch
        O->>G: diffCommits / diffCommitAgainstFirstParent
        G-->>O: changedFiles, lineStats

        alt 无变更 Java 文件
            O->>PG: updateTaskStatus(SUCCESS)
            O-->>S: result（空解析列表）
        else 有变更文件
            loop 每个 ChangedFile
                alt DELETED
                    O->>GA: listEmbeddingKeysForJavaFile
                    GA->>N: 查询
                    N-->>GA: keys
                    O->>ES: deleteEmbeddingsByNodeIds(keys)
                    O->>GA: deleteFileNodes(path)
                    GA->>N: DETACH DELETE 子图
                else MODIFIED
                    O->>GA: deleteFileOutgoingRelations(path)
                end
                opt ADDED / MODIFIED 等需重解析
                    O->>P: parse(path, sourceRoots, repoId)
                    alt 解析失败
                        O->>PG: ParseErrorPort.record
                    else 成功
                        O->>O: 收集 ParseResult
                    end
                end
            end

            opt 存在成功 ParseResult
                O->>C: batchWriteParseResults(results)
                C->>GA: batchMerge* + batchMergeRelations
                GA->>N: MERGE
            end

            O->>GA: batchMergeCommitNodes(CommitNode)
            GA->>N: MERGE Commit
            O->>CR: markProcessed(...)
            O->>PG: updateTaskStatus(SUCCESS)
            O-->>S: IncrementalIndexResult
        end
    end
```

## 4. GitHub Push Webhook：HTTP 与图谱及下游的时序

Webhook 在**同一次 HTTP 请求**内完成：验签、仓库对齐、**增量图谱**、`ChangeEvidenceAssembler`；**Dify 工作流**在 `difyWorkflowExecutor` 中异步执行，响应不等待其结束。

```mermaid
sequenceDiagram
    autonumber
    participant GH as GitHub
    participant HC as GithubWebhookController
    participant HS as GithubPushWebhookService
    participant PG as PostgreSQL<br/>(RepositoryPort)
    participant O as IncrementalIndexOrchestratorImpl
    participant CE as ChangeEvidenceAssembler
    participant Pool as difyWorkflowExecutor
    participant Dify as DifyWorkflowClient

    GH->>HC: POST /api/v1/webhooks/github<br/>(raw bytes + headers)
    HC->>HS: handlePush(rawBody, signature256, githubEvent)

    alt 非 push / 验签失败 / JSON 或仓库错误
        HS-->>HC: 204 / 401 / 400 / 404
        HC-->>GH: ResponseEntity
    else push 且仓库匹配
        HS->>PG: findActiveByCloneUrl(normalized)
        PG-->>HS: RepositoryInfo
        HS->>O: run(CommitEvent)
        Note over O: 见 §3 增量索引全链路

        alt skippedAlreadyProcessed
            O-->>HS: skippedDuplicate=true
            HS-->>HC: 200 GithubWebhookAck(indexed=false)
        else 已执行索引
            O-->>HS: IncrementalIndexResult
            HS->>CE: assemble(repoId, indexResult, ...)
            CE-->>HS: ChangeEvidenceDocument
            HS->>HS: ObjectMapper 序列化 JSON
            HS->>Pool: execute( Runnable: Dify )
            Pool-->>HS: 立即返回（不阻塞）
            HS-->>HC: 200 GithubWebhookAck(indexed=true, 统计字段)
            HC-->>GH: ResponseEntity

            Pool->>Dify: runWorkflowBlocking(evidenceJson)
            Note over Pool,Dify: 异步线程内执行；异常仅打日志
        end
    end
```

## 5. Kafka 增量消费（简图）

与 §3 共用 `IncrementalIndexOrchestrator.run`；消费成功后才 `Acknowledgment.acknowledge()`。

```mermaid
sequenceDiagram
    participant K as Spring Kafka 监听线程
    participant IC as IndexConsumer
    participant O as IncrementalIndexOrchestratorImpl

    K->>IC: onCommitEvent(CommitEvent, Ack)
    IC->>O: run(event)
    O-->>IC: ok / skip / exception
    alt 成功
        IC->>IC: acknowledgment.acknowledge()
    else 失败
        IC-->>K: throw（触发重试 / DLQ 策略由 Spring Kafka 配置决定）
    end
```

## 6. 与《图谱数据结构设计》的衔接

- 全量/增量最终都通过 `CodeGraphService` 将 `ParseResult` 中的节点与 `GraphRelation` 批量 MERGE 到 Neo4j，**实体与关系类型**见根目录 `图谱数据结构设计.md`。
- 全量额外通过 `EmbeddingService` 将本批中的 `Type`/`Method` 异步写入向量存储；增量仅在**文件删除**路径删除对应向量。
